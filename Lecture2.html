Why should we use version control?

Why should we use version control?

  • Work together without overwriting each other
  • Track every change with full history
  • Experiment safely with branches
  • Review, automate, and ship with confidence

1) Collaboration

  • Parallel work via branches
  • Merge changes instead of emailing files
  • Resolve conflicts transparently

2) History & Rollback

  • Every commit = a snapshot with author & timestamp
  • Revert bad changes quickly
  • Blame/annotate to see who changed what, when, and why

3) Branching & Experimentation

  • Try ideas without breaking main
  • Feature branches for focused work
  • Safe prototypes and spikes

4) Code Review & Quality

  • Pull requests enable discussion
  • Automated checks (lint/tests) on each commit

5) Traceability & Compliance

  • Link commits to issues and tasks
  • Audit trails for regulated environments
  • Release notes from commit history

6) Backup & Disaster Recovery

  • Cloned repos = distributed backups
  • Recover lost work with history
  • No more “final_v7_really_final.zip”

7) CI/CD & Automation

  • Build, test, and deploy on every push
  • Consistent, repeatable releases
  • Faster feedback loops

Git Basics

1. Initialize Git

  • Go to your project folder.
  • Run git init to start version control.
  • This creates a hidden .git folder.

cd my-project
git init

2. Stage and Commit

  • Stage files: git add
  • Commit with a message: git commit -m

git add index.html
git commit -m "Initial commit"

3. Add Remote

Link your local repo to GitHub (or any remote):


git remote add origin https://github.com/user/repo.git

4. Push

Send your commits to the remote branch:


git push -u origin main

5. Branching

  • Create a branch: git branch feature-x
  • Switch to branch: git checkout feature-x
  • Or combine: git checkout -b feature-x

git checkout -b feature-login

6. Merge

Merge another branch into current branch:


git checkout main
git merge feature-login

7. Example Merge Conflict

Conflict occurs when both branches modify the same line:


<<<<<<< HEAD
console.log("Hello from main");
=======
console.log("Hello from feature");
>>>>>>> feature-login

8. Resolve Merge Conflict

  • Edit file and keep correct version.
  • Stage the fixed file.
  • Commit the merge resolution.

# After fixing code
git add file.js
git commit

9. Stash & Apply

Temporarily save your changes:

  • git stash → save changes
  • git stash apply → reapply changes

git stash
git checkout main
git pull
git stash apply

Using Git through command-line can be a bit cumbersome

As an alternative, you can use a source control client like Github Desktop or Sourcetree

You can also use Source Control available in IDE like Visual Studio Code

Hands-on Task

  1. Run git init in a new folder
  2. Add hello.txt, commit
  3. Create branch feature, edit hello.txt, commit
  4. Switch back to main, edit same line, commit
  5. Merge feature into main → conflict!
  6. Edit file to fix, then git add & git commit

Feel free to use IDE (if available). Try running this command after you are done

git log --graph --oneline --all

And design, in general

What is design?

  • The definition of design vary across different domains
  • In this case,
    design is the process of creating purposeful, functional, and visually appealing solutions to meet user needs.
  • Design is (and should be) tech-stack agnostic

Designing is primarily a designer's job because:


  • it requires expertise in UI/UX principles to ensure a good user experience.
  • it requires proficiency in design tools like Figma, Adobe XD, etc.
  • it requires creativity and visual problem-solving.

A practical first step in the design process is creating wireframes.

Wireframes are simple, low-fidelity sketches or blueprints of an application’s layout that focus on structure and functionality rather than colors or visuals.

Eventually, wireframes evolve into full designs with colors, interactions, and detailed visuals, which developers then implement in code.

Wireframes Capture:

Layout and structure - placement of headers, content, buttons, and menus.

Navigation flow - how users move between screens or pages.

Content hierarchy - which elements are most important and their order.

Functional elements - forms, buttons, input fields, and interactive components.

Basic interactions - clicks, links, and transitions (without detailed animations).

Page or screen relationships - overall app or website architecture.

Wireframes are relatively easy to build, especially with modern tools at our disposal.
